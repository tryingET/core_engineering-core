# Discipline — Service/API

## Purpose

Service/API guidance owns the cross-language invariants for networked and process-boundary systems. Language lanes choose framework, validation library, ORM, queue, and deployment commands. This discipline decides what must be true before an API boundary is created, changed, operated, or retired.

## Load when

Load this discipline when work touches:

- HTTP, RPC, GraphQL, WebSocket, event, queue, webhook, plugin, FFI, CLI-as-API, or file-exchange contracts
- request/response schemas, auth, error semantics, idempotency, retries, or background jobs
- health/readiness, migrations, service deployment, observability, SLOs, or rollback
- extracting a new service from a library, worker, CLI, or monolith

Do not load it for an internal module boundary with no external consumer, no operational lifecycle, and no compatibility contract.

## When not to create a service

Do not create a service just to separate code aesthetically. Prefer a library/module, in-process adapter, scheduled job, or repo-local command when:

- there is one deployable owner and no independent scaling, trust, failure, or release boundary
- latency, auth, observability, and deployment overhead would exceed the benefit
- the proposed API would mostly proxy a database or another API without adding policy
- consumers are not yet known, or the contract is still volatile enough to change in-process
- rollback would require coordinated changes across callers anyway

Create or split a service only when it earns an explicit boundary: independent ownership, security/trust separation, scaling profile, failure isolation, external integration contract, or release cadence.

## Boundary contract

Every service/API boundary needs:

- named owner and consumer set, even if the first consumer is internal
- contract shape: endpoints/events/commands/files, versioning policy, compatibility promise, and deprecation path
- domain boundary: which concepts and invariants this API owns, and which it merely translates
- data boundary: canonical store, derived projections, cache semantics, and whether the API is allowed to write the fact
- trust model: caller identity, authn/authz, data classification, rate/abuse posture, and audit needs
- failure model: timeout, retry, idempotency, partial success, backpressure, and dead-letter behavior where relevant
- deployment model: runtime, configuration, migration order, readiness gates, rollback path, and operational dashboard/log entrypoints

If these cannot be stated, the boundary is not ready to be a shared service/API.

## Domain and data boundaries

A service boundary is not automatically a domain boundary. A service may own a domain concept, translate another bounded context, or expose a narrow integration facade. State which one it is.

- Do not let request DTOs, database rows, or vendor payloads become the domain model by accident.
- Put domain invariants where every write path must pass, or prove equivalent enforcement for each path.
- Do not expose direct CRUD over data that has lifecycle, permission, workflow, or consistency rules.
- Avoid two writers for the same fact; if two writers are unavoidable, define conflict and reconciliation policy.
- Treat events as completed domain facts, not arbitrary implementation notifications, when downstream consumers rely on them.

Cross-reference `domain-modeling` for vocabulary/invariants/workflows and `data-governance` for authority/schema/lineage/lifecycle.

## Request/response and schema invariants

- Validate untrusted input at the boundary, not only after business logic starts.
- Validate output when consumers depend on a published contract or generated schema.
- Make required/optional/null/default behavior explicit; do not rely on framework accidents.
- Treat schema evolution as compatibility work: additive changes by default, removals and semantic changes behind versioning or migration plans.
- Keep generated schemas reproducible and checked or regenerated by deterministic commands.
- Separate transport errors from domain errors; both need stable machine-readable shape.

Use the lane-native schema tooling, but keep the invariant language-independent: external data is parsed, normalized, authorized, and only then trusted.

## Auth, authorization, and tenancy

- Authentication identifies the caller; authorization decides the action and resource. Do not collapse them.
- Enforce authorization server-side at the domain/action boundary, not only in UI or routing.
- Record tenant/user/resource identifiers in audit-safe logs and traces without leaking secrets or sensitive payloads.
- Default-deny unknown scopes, roles, tenants, and resource states.
- For machine-to-machine APIs, define token audience, rotation, expiry, and revocation posture.

Cross-reference `security-privacy` for secrets, data classification, privacy, telemetry redaction, and dependency risk.

## Error, retry, and idempotency

- Errors must be stable enough for callers to handle: include machine code, human-safe message, retryability, and correlation/request id where useful.
- Do not expose stack traces, raw SQL, secret-bearing config, or internal topology to callers.
- Retried writes need an idempotency key, natural idempotent resource identity, or documented non-idempotent semantics.
- Background jobs and webhooks need dedupe, retry budget, poison/dead-letter handling, and operator visibility.
- Timeouts are part of the contract; infinite waits are bugs.

## Background jobs and async boundaries

Use background jobs/events when work is slow, bursty, retryable, scheduled, fan-out, or needs failure isolation. Do not hide user-visible synchronous requirements behind a queue without a status/read model.

Async boundaries need:

- durable payload schema and versioning
- idempotent handlers or dedupe keys
- retry/backoff/dead-letter policy
- observability for enqueue, start, success, failure, retry, age, and backlog
- replay/migration behavior when payload schema changes

## Data and migrations

- Schema/data migrations are deployment work, not incidental code changes.
- Prefer expand/contract migrations for compatible rolling deploys.
- Record ordering: migrate before/after deploy, dual-write/read period, backfill, cleanup, and rollback implications.
- Avoid service APIs that bypass data authority or create two writers without conflict policy.
- Cross-reference `local-first-data` when local stores, projections, imports/exports, or sync are involved.

## Health, readiness, and operations

A service should expose or document:

- liveness: process/event loop is alive
- readiness: can serve real traffic, including critical dependencies and migrations where appropriate
- startup/shutdown: graceful drain, queue worker stop behavior, and connection cleanup
- operational runbook entrypoints: logs, traces, metrics, dashboards, and common failure modes

Health checks must not become expensive synthetic transactions unless the service explicitly owns that cost.

## Observability and SLO posture

Instrument stable boundaries:

- request/job/event count, latency, error rate, saturation/backlog, retry/dead-letter counts
- dependency calls with timeouts and error classes
- correlation/request ids across service and job boundaries
- structured logs for state transitions and failure decisions

Use `observability` for signal design and `performance` for latency/throughput/memory budgets. Do not claim an SLO without measurement and an owner for the error budget.

## Deployment and rollback

Before release, know:

- artifact identity and config source
- compatibility with current callers, databases, queues, and workers
- migration order and rollback safety
- feature flag or traffic-shaping posture for risky behavior changes
- validation command/evidence for smoke, fast, and full tiers

Rollback must consider code, config, schema, queued payloads, generated clients, and data already written under the new version.

## Validation expectations

Minimum evidence scales with risk, but service/API changes commonly need:

- contract tests or schema generation checks for public/consumer-facing boundaries
- authz tests for protected actions and tenant/resource separation
- idempotency/retry tests for write and job boundaries
- migration test or dry run for schema/data changes
- health/readiness smoke where deploy/runtime behavior changed
- observability proof when adding a new operational boundary

Use `validation`, `testing`, `specification-and-dsls`, `domain-modeling`, `data-governance`, `observability`, `security-privacy`, and `release-package` as applicable.

## Failure modes

- A "service" that is only a remote function call with worse latency and harder rollback.
- Schema types exist in code but untrusted input is not validated at runtime.
- Transport DTOs, ORM rows, or vendor payloads silently become the domain model.
- Error strings become accidental API contracts.
- Retries duplicate payments, emails, jobs, imports, or destructive actions.
- Readiness goes green before migrations/config/dependencies are actually usable.
- Queue payload changes strand old jobs.
- Rollback plan ignores data already migrated or events already emitted.
