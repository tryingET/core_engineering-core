# Local release draft — engineering-core v0.3.3

## Title

`engineering-core v0.3.3 — adoption scanner and cross-scope visibility`

## Release body

This patch release adds the reusable `engineering-core scan-adoption` command and scanner modules. Engineering-core now owns generic adoption scanner semantics while scanned scopes keep their generated dashboards, JSON snapshots, and rollout interpretation.

The scanner supports single-repo, lane-root, company-root, and workspace-style scopes, including `~/ai-society/core`, `softwareco/owned`, `softwareco/infra`, `holdingco`, `teachingco`, and `healthco`.

### Breaking Changes

- None.

### Added

- `engineering-core scan-adoption` CLI command.
- Reusable adoption scanner and markdown renderer modules:
  - `src/engineering_core/adoption_scan.py`
  - `src/engineering_core/adoption_render.py`
- Structural adoption statuses for adopted, partial, doc-only, policy-only, legacy-only, legacy-mixed, invalid-policy, and missing surfaces.
- Catalog-aware lane and discipline validation.
- Multi-scope and recursive repo discovery support.
- Package/member adoption surface discovery from current and legacy docs/policies.
- `docs/vision.md` describing engineering-core as the shared guidance and adoption-visibility substrate.
- Tests for scanner classification, package discovery, invalid policy handling, CLI JSON output, and catalog version consistency.

### Changed

- Documented the owner boundary: engineering-core owns scanner semantics; scope owners own generated scan outputs and rollout dashboards.
- Updated README, adoption docs, authority map, and AGENTS guidance for scanner ownership and release validation.
- Bumped package/catalog metadata to `0.3.3`.

### Fixed

- Scanner behavior rejects valid JSON with the wrong top-level type as `invalid-policy` instead of crashing or silently treating it as missing.
- Single-repo scans now auto-include the scope root when appropriate.
- Scanner fails on missing scopes instead of returning a misleading empty success result.
- Core semantic heuristics avoid hardcoded company/repo-specific tokens.

### Agent migration notes

- Existing commands such as `show`, `show-discipline`, `show-all-for`, and `recommend` are unchanged.
- Lane/company wrappers should delegate to `engineering-core scan-adoption` instead of keeping forked scanner logic.
- Generated outputs such as `governance/engineering-core-adoption-scan.json` and `docs/project/engineering-core-adoption-dashboard.md` should remain in the scanned scope.
- Use warning/ratchet CI before hard adoption gates while scope owners classify intentional local posture.
